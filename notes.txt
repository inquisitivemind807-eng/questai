### 1. Login Page: Persistent Loading & Error Messages
*   **Problem:** Email/password not found error message is missing. A loading
    screen shows up every time, even on fast machines.
*   **Finding:** The "stupid loading screen" is caused by the global
    `$authService.loading` state in `src/routes/+layout.svelte`, which is `true`
    by default. It only becomes `false` after `authService.initialize()`
    completes its async operations (reading from cache via Tauri `invoke`).
    Error messages currently pass through from the API via `handleSubmit` in
    `src/routes/login/+page.svelte`; we should map `401` or specific error
    strings to "Email or password not found" instead of showing generic API
    errors.
*   **Proposed Solution:** Map `401` and specific error strings to a friendly
    "Email or password not found" message in `handleSubmit`. Modify
    `$authService.loading` to handle fast loads more gracefully without a
    prominent loading spinner.

### 2. Payment Redirect: Session Loss
*   **Problem:** Clicking "Back" from the sandbox payment page redirects to the
    login page (logged out).
*   **Finding:** The logout is triggered by `authService.initialize()` in
    `src/lib/authService.js`. When you return from the sandbox (an external
    origin), the app re-initializes. If "Remember Me" was not checked, the
    `auth_remember_me.txt` flag is 'false', causing `initialize()` to
    aggressively wipe all tokens and the session, effectively logging you out
    upon your return.
*   **Proposed Solution:** Update `initialize()` to not wipe tokens aggressively
    during an active session navigation, perhaps keeping the session alive in
    `sessionStorage` or avoiding immediate cache clearing just because
    `rememberMe` is false.

### 3. Bot Logs: Duplicate Panels
*   **Problem:** Bot dashboard shows two separate log panels for the same bot.
*   **Finding:** In `src/lib/stores/botProgressStore.ts`, the store uses `botId`
    as the key. When a bot starts, it might initially use a generic name (e.g.,
    'seek'), but then a `[BOT_EVENT]` might arrive with a specific `botId`
    (e.g., 'seek_123'). If the logic doesn't unify these, `addLogLine` creates a
    new entry in the `bots` record, leading to duplicate panels in
    `src/routes/bot-logs/+page.svelte`.
*   **Proposed Solution:** Ensure consistent mapping between the initial generic
    `botId` and the specific event `botId` by resolving and unifying the keys
    in `botProgressStore.ts`.

### 20. Tauri Dev Error: Session/Driver Crash
*   **Problem:** SessionNotCreatedError during `bun run tauri dev`.
*   **Finding:** The `SessionNotCreatedError` occurred because ChromeDriver
    failed to start a new Chrome instance. This is often caused by a stale
    `LOCK` file in the user data directory (e.g., `sessions/seek/LOCK`). While
    `src/bots/core/browser_manager.ts` has `cleanStaleLockFiles`, it only
    removes locks older than 1 hour. If a previous crash was recent, the bot
    fails to start. Additionally, the "invalid session ID" errors indicate the
    overlay system tried to update after the driver had already crashed or been
    closed.
*   **Proposed Solution:** Modify `cleanStaleLockFiles` in `browser_manager.ts`
    to forcefully delete the `LOCK` file regardless of its age upon initializing
    a new bot run to prevent `SessionNotCreatedError`.

### 21. Authentication: Refresh Token Logic (Deep Dive)
*   **Finding:** The access token's 15-minute lifespan is the root of the
    problem. `authService.initialize()` runs on every page load and manually
    checks this expiry; if expired, it calls `logout()`, which wipes
    everything—including the refresh token and the "Remember Me" flag. This
    prevents `CorpusRagAuth.validateSession()` from ever attempting a refresh.
    Furthermore, `initialize()` aggressively deletes sessions on refresh if
    "Remember Me" was unchecked.
*   **Summary of what's happening:**
    1. You login (with or without Remember Me).
    2. 15 minutes pass. Your Access Token expires.
    3. You refresh the page or come back to the app.
    4. `initialize()` runs, sees the expired token, and calls `logout()`.
    5. `logout()` deletes everything from your disk/localStorage.
    6. You are redirected to the Login page.
*   **Proposed Fix Strategy:**
    1. Modify `initialize()` in `src/lib/authService.js` to attempt a token
       refresh if the access token is expired, instead of calling `logout()`.
    2. Update `logout()` to be more selective, or ensure it's not called
       prematurely.
    3. Adjust "Remember Me" handling so that 'false' doesn't trigger an
       immediate deletion of the current active session during a refresh.
*   **Proposed Solution:** Explicitly call `refreshAccessToken()` inside
    `initialize()` if the token is found to be expired before executing the
    `logout()` sequence, ensuring session continuity.
